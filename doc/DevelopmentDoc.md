# 自动标注模块开发需求

## 文档状态

- 文档类型：项目级开发需求
- 当前阶段：MVP 设计
- MVP 范围：串行跑通完整自动质检与标注流程
- 测试输入：`tests/train_data_1.mcap`

## 开发目标

构建一个机器人 VLA 项目数据自动质检与标注服务，用于 VLA 训练数据的清洗和准备。

服务需要对来源不一、格式不定、采集方式多样的机器人视频、图像和日志数据进行解析、质检、事件划分和事件标注，最终输出可供训练数据清洗、人工复核和后续数据转换使用的结构化结果。

第一版目标如下：

1. 输入解析：解析视频、图像和机器人日志数据，统一成后续模块可消费的结构化时间序列。
2. Topic 配置输入：MVP 阶段不开发 Topic 检测模块，camera、state、action 等 topic 信息由外部请求或配置显式输入。
3. 数据质检：检测图像黑帧、模糊帧、损坏帧、静止帧，以及机器人数据异常跳变、动作与状态不一致、运动超限等问题。
4. 事件生成：基于外部配置指定的机器人 topic 数据生成事件节点。
5. 事件标注：将事件片段中的视频图像传入 VLM 模型，结合 prompt 输出事件标签和描述。

## 非目标

MVP 阶段暂不要求：

- 并发调度和任务抢占。
- 多文件格式完整覆盖。
- 性能优化和大规模吞吐指标。
- 人工标注平台交互闭环。
- 自动修复低质量数据。
- Topic 检测模块开发；MVP 阶段 topic 信息必须由外部请求或配置输入。

## 执行流程

1. 输入解析阶段：对传入的视频、图像和机器人数据进行解析、排序、同步和对齐，生成后续处理需要的数据结构。
2. Topic 配置输入阶段：从外部请求或配置中读取 camera、state、action 等 topic 信息；MVP 阶段不根据机器人数据自动推理 topic。
3. 数据质检阶段：对机器人数据进行运动突变检测、极值超限检测；对视频和图像进行质量检测；合并相邻低质量片段并输出异常记录。
4. 事件生成阶段：基于外部配置指定的机器人 topic 数据生成机器人运动事件节点。
5. 事件标注阶段：将事件节点之间的视频图像传入 VLM 模型，结合 prompt 生成事件标签和描述。

## 并发与过程控制

第一版不要求并发控制，全流程串行跑通即可。

后续引入并发控制时，由服务编排层生成运行时字段，例如 `job_id`、子任务 ID、重试次数、任务状态等，并将这些字段注入各模块请求。

## 配置层级说明

服务入口配置是全流程编排层配置，包含全局字段和各模块字段。

模块级配置与服务级配置可能结构相似，但二者层级不同：

- 服务级配置描述一次完整自动标注任务的请求。
- 模块级配置描述单个模块实际运行时需要的请求。
- 编排层负责将全局字段、运行时生成字段和模块配置组合成模块请求。

例如 `task_id` 属于全局字段，需要被多个模块复用；未来的 `job_id` 可能由服务在请求到达后生成，也不应强行写入某个单独模块配置内部。

## 服务入口配置

服务入口配置为 JSON 格式，基本结构如下：

```json
{
  "basic_config": {
    "task_id": "自动生成或外部传入的任务 ID"
  },
  "parser_config": {},
  "topic_config": {
    "camera": {},
    "state": [],
    "action": []
  },
  "data_check_config": {},
  "event_generation_config": {},
  "event_labeling_config": {}
}
```

字段说明：

- `basic_config`：全流程共享字段，例如 `task_id`，未来可扩展 `job_id`、运行环境、用户信息等。
- `parser_config`：输入解析模块配置。
- `topic_config`：外部输入的 topic 配置，MVP 阶段由请求方显式提供，不由服务自动检测生成。
- `data_check_config`：数据质检模块配置。
- `event_generation_config`：事件生成模块配置。
- `event_labeling_config`：事件标注模块配置。

## 模块请求组装原则

模块执行时，编排层按以下原则组装请求：

1. 全局字段来自 `basic_config`。
2. 模块私有字段来自对应模块配置。
3. Topic 信息来自外部输入的 `topic_config`，由编排层注入 Parser、数据质检、事件生成等需要 topic 的模块请求。
4. 运行时字段由编排层生成，例如 `job_id`、临时目录、下载后的本地路径等。
5. 模块之间只通过明确的结构化产物传递数据，不直接读取彼此内部状态。

Parser 模块请求示例：

```json
{
  "basic": {
    "task_id": "TASK-001",
    "job_id": "JOB-001"
  },
  "parser": {},
  "align": {},
  "insert": {},
  "output_format": {}
}
```

## MVP 验收标准

MVP 阶段以“可重复跑通完整流程”为验收目标。

必须满足：

1. 使用 `tests/train_data_1.mcap` 作为输入可以完成一次完整流程。
2. Parser 模块能识别配置中的 camera、state、action topic。
3. Parser 模块能输出按主时间轴对齐的 `timestamp`、`image`、`state`、`action` 数据。
4. 输出 timestamp 单调递增。
5. 必需 topic 未在外部配置中提供时失败；可选 topic 缺失时按模块配置处理。
6. 数据质检模块能输出结构化异常记录，即使没有异常也要输出空结果。
7. 事件生成模块能基于质检异常区间输出事件节点列表。
8. 事件标注模块能输出每个事件的标签和描述字段。
9. 全流程运行失败时能定位到失败模块、文件、topic 和错误原因。
